Authenticate via OAuth

  • By default, only HTTP Auth is enabled. To enable OAuth, have your administrator set the 'OAUTH_ENABLED' flag to true in settings.py
  • While registering, submit the name and description of the client you wish to authorize
  • Once submitted, you will receive a client ID and client secret. Use these with the appropriate OAuth headers to perform the initial handshake to gain your temporary token. ({% url oauth:oauth_provider.views.request_token %}) NOTE: If you are not familiar with OAuth, please consult the spec and the OAuth 1.0 RFC
  • You'll then receive your request token. Use this when visiting the authentication page ({% url oauth:oauth_provider.views.user_authorization %}). An LRS OAuth ID will be issued to you for you to POST back to the authentication page from your client
  • Once the client is authenticated, send the appropriate OAuth headers and values from your token to gain an access token.({% url oauth:oauth_provider.views.access_token %})
  • You can now send this access token in with the OAuth headers when making calls to the LRS system

Authenticate via OAuth2

  • By default, only HTTP Auth is enabled. To enable OAuth, have your administrator set the 'OAUTH_ENABLED' flag to true in settings.py
  • While registering, submit the name, client url, redirect url, and type of the client you wish to authorize
  • Once submitted, you will receive a client ID and client secret. Use these with the appropriate OAuth2 headers to authorize. (/XAPI/oauth2/authorize) NOTE: If you are not familiar with OAuth2, please consult the the OAuth 2.0 RFC
  • Once the client is authenticated, send the appropriate OAuth2 headers and values from your token to gain an access token.(/XAPI/oauth2/access_token)
  • You can now send this access token in with the OAuth2 headers when making calls to the LRS system

Registration is used to enable HTTP Basic Authentication. Without using secure sockets (SSL), this authentication scheme is likely not secure enough for most production systems. Please do not register using sensitive names and passwords. Assume all data provided here is public. After you register a user, you can use the HTTP_AUTHORIZATION header with your encoded username and password.
